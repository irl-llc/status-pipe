The product sentence: **"All quiet — 3 agents running, 2 done today."** Every
layout decision serves getting the operator to that sentence, or to the next
item that needs them, as fast as possible. The view is a queue, not a dashboard.

## Visual language

These rules bind every component; wireframes and snapshot tests are checked
against them:

1. **Codicons only.** Every icon is a codicon (`@vscode/codicons`, already in
   the build); where no codicon fits, we draw a custom SVG glyph in the same
   16×16, stroke-style family. No emoji anywhere.
2. **One indicator per fact.** A labeled button carries no icon; a
   color-accented banner carries no additional warning glyph; a kind icon next
   to "review" is the indicator — the word is the label, not a second
   indicator.
3. **Non-default status only.** Indicators appear when there is something to
   say: passing CI shows nothing (detail on hover), failing/pending CI shows;
   comment counts show only when unresolved > 0; tasks only when unresolved
   > 0; draft only when draft; review decision only when it isn't the neutral
   default. A healthy PR row is just `#512 T1a rate-limit-core`.
4. **Detail on hover.** Tooltips carry the full story (per-check list, full
   comment breakdown, exact timestamps, full headline); secondary actions
   (withdraw ack, open log, open ticket file, per-row buttons) appear on row
   hover. Omnipresent elements are limited to what triage needs at a glance:
   accent color, identity, headline, the waiting line, primary action.
5. **No layout shift.** Indicators that come and go (activity spinner,
   staleness, backoff) live in **permanently reserved slots** — they change
   contents, never geometry. Nothing in the queue moves because a background
   refresh started. (The git-spice extension's jumping spinner is the named
   anti-pattern.)
6. **Meaning is never JSON-only.** Every encoded indicator — an accent color, a
   status codicon, a staleness tint — carries a tooltip naming the fact in
   words ("Blocked — needs you", "Stale worker — no heartbeat in 40m"). The
   color/glyph is the at-a-glance signal; the word behind it is one hover away.
   What a color or icon *means* must never require opening the ticket file.

### The activity & staleness slot

One reserved 16×16 slot at the far right of each view's header row (tray and
editor tab), plus a status-bar item in editor mode:

- idle: empty (slot reserved, so nothing moves)
- background refresh in flight: `sync~spin`
- degraded (rate-limited / offline / auth): `warning` tinted amber, tooltip
  with cause, data age, retry time, and a "retry now" link
- click: force refresh of everything in the view's scope

This is the **only** place global forge/network status appears. Cards show at
most a staleness tint; per-repo detail lives on the repo header's hover.

## Queue semantics

Three lanes, computed per ticket from protocol state ∪ forge enrichment. The
underlying predicate for the first lane is *"agent parked, AND parked on me."*

### NEEDS YOU — any of:

1. Launcher failed (fleet mode): the repo's supervisor exhausted restarts —
   synthetic repo-level card ([09-launch-and-supervision.md](09-launch-and-supervision.md))
2. `worker.status = error`, or `worker.status = running` with
   `now − heartbeatAt > staleWorkerMinutes` (crashed/stale agent)
3. `health = blocked` or `blockers[]` non-empty
4. `waitingOn.kind = owner`, or `waitingOn.kind = comment` (a reply is
   awaited and the operator is the default responder; if the awaited reply is
   attributably someone else's, demote to Waiting)
5. `waitingOn.kind = review` (when the forge can attribute the requested
   reviewer and it isn't the local user, demote to Waiting; if unattributable,
   include — false positives beat silent starvation)
6. `waitingOn.kind = merge` (human-gated merge)
7. Orphaned failing CI: any PR `ci = failing` **and** `worker.status = idle`
   **and** `phase ∉ {fixing}` — red CI that nobody is working
8. Stale ack: a pending ack that an orchestrator pass completed *after* without
   consuming, or — when no pass has run since it — older than 2 × the expected
   tick interval (full rule in [02-protocol.md](02-protocol.md)); the operator
   signaled and the orchestrator isn't picking it up

### WAITING ON WORLD

(The rendered section label, everywhere it appears — tray, editor pane,
docs — is `WAITING ON WORLD`.)

Agent running with fresh heartbeat · `waitingOn.kind = build` with CI pending ·
review requested from someone else · merged-on-forge awaiting agent catch-up ·
ack sent and within the pickup window.

### QUIET

`phase ∈ {merged, abandoned}` or `health = done`. Shown with completion
timestamp for 24h (configurable), then hidden behind the "show done" filter.

### Priority order within NEEDS YOU

1. Launcher failed / crashed / stale / errored workers — a dead orchestrator or
   agent generates no further signal; until restarted, nothing downstream
   recovers
2. Stale acks — same logic: infrastructure failure masquerading as progress
3. `health = blocked` (explicit owner-must-act)
4. `waitingOn.kind ∈ {owner, comment}` (questions/replies; agent fully parked)
5. Review requests (agent parked, but the work product is batch-reviewable)
6. Merge actions (one click; work already finished)
7. Orphaned failing CI

Tie-break: oldest `waitingOn.since` first (newest-first starves the long tail,
and with parallel agents the tail is where throughput dies), then repo name,
then ticket key — fully deterministic so cards never shuffle between renders.
A blended "staleness score" was considered and rejected: opaque scores make the
queue unauditable; (class, age) is explainable at a glance.

## Card anatomy (tray)

See [wireframes/card-anatomy.svg](wireframes/card-anatomy.svg). Omnipresent
parts are deliberately few; everything else is hover/expand.

- **Accent bar** (left edge, 3px): health color — red `blocked/error`, amber
  `waiting`, blue `ok/running`, green `done`. This *is* the health indicator;
  no dot, no second glyph.
- **Header row**: repo badge (fleet mode only; hidden in single-repo mode) ·
  `#142` · title (1 line, ellipsis) · relative age. Phase is shown as plain
  dim text, not a chip — it's context, not status.
- **Headline**: 2-line clamp, newlines/markdown stripped; full text on hover.
  The agent's own sentence, verbatim.
- **Waiting line** (when `waitingOn ≠ null`): kind codicon + `detail` (1 line)
  + live duration, amber text — a text line, not a filled banner.
- **Blockers**: red text lines, verbatim, never truncated. The red is the
  indicator; no icon.
- **PR rows** (open PRs first; merged collapse to a single dim
  `3 merged` line, expandable):
  - upstream indicator above, small/dim: `↑ main` or `↑ T1a #855`
  - `#855` · part chip `T1a` · head branch (dim) — and **only non-default
    badges**: CI `x`/`clock`, unresolved comments, unresolved tasks, draft,
    changes-requested, linked ticket. A green PR row carries no badges; hover
    reveals the full breakdown (checks list, totals incl. resolvable
    captioning).
  - downstream indicator below, small/dim: `↓ T2 #861, T3 #870`
- **Action row**: one contextual primary text button — `Open question` /
  `Open PR` / `Restart worker` — plus a fixed second slot for the **ack
  control** (`Ready for another look` button when actionable; the sent chip
  once acked). No icons on buttons. The ack control gets a permanent slot
  rather than hover treatment because it is the product's core interaction;
  its withdraw/note-edit affordances appear on hover.

### Ack'd card — the calm state

When the operator hands a card back (a `pending` or `picked-up` ack) **and**
the ack actually moved the card out of NEEDS YOU, the card reads as *done on
your end, awaiting pickup* — it must not keep alarming as if you still owe it
an answer:

- **Accent bar** turns calm blue (`charts-blue`), overriding the health color.
- **Waiting line and blockers** drop to `descriptionForeground` — they are now
  stale context, not a call to action. Their meaning stays on hover (visual
  rule #6); the text itself never disappears.
- **Ordering**: within WAITING the ack'd card sinks *below* un-ack'd waiting
  work, so the operator's already-handled item stops sitting at the top.

A pending/picked-up ack on a card that is **still** in NEEDS YOU (a crashed or
stale worker, a `stale`/`pickup-unconfirmed`/`superseded` ack, or a re-looked
question still waiting on the owner) keeps its alarm — the calm state is only
for cards the ack genuinely parked.

## Side tray (compact triage index)

- **Header** (one fixed row): title · needs-you count · filter icon (expands a
  text box in place of the title, Esc restores) · overflow menu (group-by-repo,
  show-done, sort) · the reserved activity slot.
- **Agents strip** (fleet mode; single collapsed row, fixed height):
  `agents: 2 running · 1 scheduled 3m · 1 failed` — expands to one row per
  (repo, agent) with status codicon, next/elapsed time, and hover actions
  (start/stop/tick-now/open-log). Per-agent states include `parked — all work
  waiting on you` (orchestrator-declared; ticks suspended until an ack or
  backlog change). See
  [09-launch-and-supervision.md](09-launch-and-supervision.md).
- Lanes as collapsible sections with counts: `NEEDS YOU (4)`, `WAITING (2)`,
  `QUIET (1)`.
- Default is one merged queue with per-card repo badges (cross-repo severity
  ordering is the point); group-by-repo is a toggle.
- Empty needs-you state: the product sentence, centered. Its inverse — needs-you
  non-empty with nothing in flight and agents parked — reads "Parked — 4 items
  need you, nothing in flight."
- Inactive-roots footer (collapsed), and "orchestrator last ran" per repo on
  the repo header hover — not omnipresent.
- 30+ items: list virtualization; QUIET collapsed by default; the tray is a
  triage index — filtering gets faster, cards do not get richer.

## Editor tab (expanded workspace)

`statusPipe.openInEditor`. Two-pane: the same queue (narrower, denser) on the
left; the selected ticket expanded on the right.

**The right pane always reflects the operator's actual selection.** When the
queue updates underneath it, a selection whose card has left the set is cleared
or re-pointed — never silently swapped for an unrelated card. When nothing is
explicitly selected the pane shows the top NEEDS-YOU item *and says so* (a "most
urgent — nothing selected" affordance), so the pane can never read as
authoritative for a ticket the operator was looking at elsewhere. (The silent
fall-back to "first card" is the named anti-pattern: it makes the pane disagree
with the file the operator has open.)

**Read top-to-bottom as a hierarchy**, every block under a labeled section
header — a bare paragraph of headline followed by unheadered red blocker lines
is the anti-pattern that makes the pane denser than the raw JSON:

- **Status** — one line: health (named, not just the accent), phase, age,
  worker liveness (`worker` status + heartbeat age + orchestrator pass info —
  supervisor process state shown separately in the agents strip, labeled
  distinctly).
- **What needs you** — blockers and the waiting line, under their own header.
  Each blocker is one terse line ([02-protocol.md](02-protocol.md) brevity
  contract); a long one clamps with full text on hover/expand, exactly like the
  card — the detail pane structures verbose content, it does not dump it.
- **Context** — the headline, sectioned and markdown-stripped, not a floating
  `<p>`.
- **Evidence** — the heavier sections, each under its own header:
  - PR table: one row per PR — branch + stack refs, checks summary (failing
    checks named inline; passing collapsed to a count), unresolved/total
    comments with resolvable captioning, tasks (column present only when the
    forge has the capability), review decision, linked tickets
  - sub-tickets (epics): one row per `subTickets[]` entry — topic, status, deep
    link ([02-protocol.md](02-protocol.md)); section absent when the epic has none
  - history timeline: every `history[]` entry (time, phase, note; `runId`
    linked when present); ack consumptions/supersessions appear here
  - ack log (pending + recent), raw-JSON peek, open ticket file / epic file links

The same reserved activity slot lives in the header; plus a status-bar item.

## Click targets

| Target | Action |
|---|---|
| Waiting line / `Open question` | `waitingOn.ref` in browser — deep link to the exact comment; the highest-value click in the product |
| Ticket key / title | tracking ticket in browser (GitHub or Jira) |
| PR row | PR in browser (setting to prefer the GitHub PRs extension) |
| CI badge | failing check's URL, else PR checks tab |
| Card body | expand (tray: in-place accordion; editor: right pane) |
| Repo badge | filter queue to repo |
| Restart worker | supervisor tick-now when a launch file exists; else `statusPipe.resumeCommand` in the integrated terminal |
| Activity slot | force refresh (scope: the view); tooltip carries degraded-state detail |

## Notifications

- **Activity-bar badge = NEEDS YOU count only.** Waiting/quiet churn never
  badges.
- Toasts, individually toggleable, all default-on: new blocker or owner
  question · agent crash / stale heartbeat / stale ack / launcher failure ·
  item completed (quiet style) · CI flipped to failing with idle agent. Never
  for WAITING-lane churn. Global do-not-disturb setting.

## Deliberate omissions

No merge/approve buttons in the tray — those decisions belong on the forge page
where the full context lives (the alternative saves one click and invites
approving a stack you haven't looked at). No stack graph beyond the up/down
indicators — the git-spice extension owns stack visualization.

## Implementation status (2026-06-11)

Implemented and verified against this document except the following, all
deferred deliberately (none change the protocol or stored state):

- **Group-by-repo + sort** in the overflow menu: `ViewOptions.groupByRepo`
  is plumbed end-to-end but no menu renders it; the merged queue with repo
  badges is the only ordering.
- **Comment-demotion** ("awaited reply is attributably someone else's →
  demote to Waiting"): requires per-comment attribution data the enricher
  does not fetch; `comment` currently always escalates as `owner` (the
  conservative direction).
- **Editor ack log** (pending + recent): the detail pane shows the single
  most recent ack chip only.
- **CI badge → PR checks-tab fallback**: the failing badge deep-links only
  when a failing check carries a URL; the pending badge is not clickable.
- **Tray in-place accordion**: cards always render fully; card click only
  tracks selection (the editor's right pane uses it; the tray shows no
  visual change).
- **List virtualization at 30+ items** and the **inactive-roots footer**.
- **Repo-mismatch hover warning** (ticket file `repo` vs git remote).

The parked summary line, the merged/closed collapse split, the fact icons
(blocked `circle-slash`, done `pass-filled`, crashed `warning`), and
deterministic codepoint tie-breaks were implemented 2026-06-11 after the
adversarial review.

### Legibility revision (2026-06-24)

The brevity contract ([02-protocol.md](02-protocol.md)), the
meaning-is-never-JSON-only rule (visual language #6), the detail-pane hierarchy
(Status / What needs you / Context / Evidence), and right-pane selection honesty
are newly specified here and **not yet implemented**; they are tracked in
GitHub issues. The original cause was agents writing wall-of-text fields plus a
detail pane that flattened the ticket's structure into unheadered prose — so the
expanded view read denser than the raw JSON it was meant to beat.

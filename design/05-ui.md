The product sentence: **"All quiet — 3 agents running, 2 done today."** Every
layout decision serves getting the operator to that sentence, or to the next
item that needs them, as fast as possible. The view is a queue, not a dashboard.

## Queue semantics

Three lanes, computed per issue from file state ∪ forge enrichment. The
underlying predicate for the first lane is *"agent parked, AND parked on me."*

### NEEDS YOU — any of:

1. `run.status = error`, or `run.status = running` with
   `now − heartbeatAt > staleRunMinutes` (crashed/stale agent)
2. `health = blocked` or `blockers[]` non-empty
3. `waitingOn.kind = owner`
4. `waitingOn.kind = review` (when the forge can attribute the requested
   reviewer and it isn't the local user, demote to Waiting; if unattributable,
   include — false positives beat silent starvation)
5. `waitingOn.kind = merge` (human-gated merge)
6. Orphaned failing CI: any PR `ci = failing` **and** `run.status = idle`
   **and** `phase ∉ {fixing}` — red CI that nobody is working
7. Stale ack: a pending ack file older than 2 × `staleRunMinutes` (the
   operator signaled; the orchestrator never picked it up)

### WAITING (on the world)

Agent running with fresh heartbeat · `waitingOn.kind = build` with CI pending ·
review requested from someone else · merged-on-forge awaiting agent catch-up ·
ack sent and within the pickup window.

### QUIET

`phase ∈ {merged, abandoned}` or `health = done`. Shown with completion
timestamp for 24h (configurable), then hidden behind the "show done" filter.

### Priority order within NEEDS YOU

1. Crashed / stale / errored runs — a dead agent generates no further signal;
   until restarted, nothing downstream recovers
2. Stale acks — same logic: infrastructure failure masquerading as progress
3. `health = blocked` (explicit owner-must-act)
4. `waitingOn.kind = owner` (questions; agent fully parked)
5. Review requests (agent parked, but the work product is batch-reviewable)
6. Merge actions (one click; work already finished)
7. Orphaned failing CI

Tie-break: oldest `waitingOn.since` first (newest-first starves the long tail,
and with parallel agents the tail is where throughput dies), then repo name,
then issue number — fully deterministic so cards never shuffle between renders.
A blended "staleness score" was considered and rejected: opaque scores make the
queue unauditable; (class, age) is explainable at a glance.

## Card anatomy (tray)

See [wireframes/card-anatomy.svg](wireframes/card-anatomy.svg).

- **Accent bar** (left edge, 3px): health color — red `blocked/error`, amber
  `waiting`, blue `ok/running`, green `done`.
- **Header row**: repo badge (color-coded per repo) · `#142` · title
  (1 line, ellipsis) · phase chip · relative age.
- **Headline**: 2-line clamp, newlines/markdown stripped; tooltip carries full
  text. This is `headline` verbatim — the agent's own sentence.
- **Waiting banner** (when `waitingOn ≠ null`): kind icon (❓ owner /
  👀 review / 🔁 comment / 🔨 build / 🔀 merge) · `detail` (1 line) · live
  duration "waiting 14h" (amber > 4h, red > 24h).
- **Blockers**: verbatim bullet list, red text, never truncated.
- **PR rows** (one per `prs[]` entry, open PRs first):
  - upstream indicator above the row, small/dim type: `↑ main` or `↑ T1a #855`
  - `#855` · part chip `T1a` · draft marker · head branch (dim)
  - badges: CI dot (✓/✗/●/–) with check-count tooltip · comments `5 (2 of 3
    unresolved)` honoring `prLevelResolvable` captioning · tasks `1/4`
    (Bitbucket only) · review decision icon · linked issues `→ #91`
  - downstream indicator below, small/dim type: `↓ T2 #861, T3 #870`
- **Action row**: exactly one contextual primary action — **Ready for another
  look** (with optional note popover) / **Open question** / **Open PR** /
  **Restart run** — plus the ack chip when present.

## Side tray (compact triage index)

- Header: needs-you count, staleness chip ("forge data 12m old · rate limited"
  when degraded), refresh button, filter text box, overflow menu (group-by-repo
  toggle, show-done toggle, sort override).
- Lanes as collapsible sections with counts: `NEEDS YOU (4)`, `WAITING (2)`,
  `QUIET (1)`.
- Repo section headers only in group-by-repo mode; default is one merged queue
  with per-card repo badges (cross-repo severity ordering is the point).
- Empty needs-you state: the product sentence, centered: *"All quiet — 3 agents
  running, 2 done today."*
- Inactive-roots footer (collapsed): workspace folders without `.autopilot/`,
  so "no agents here" is distinguishable from "extension broken".
- 30+ items: list virtualization; QUIET collapsed by default; the tray is a
  triage index — filtering gets faster, cards do not get richer.

## Editor tab (expanded workspace)

`statusPipe.openInEditor`. Two-pane: the same queue (narrower, denser) on the
left; the selected issue expanded on the right:

- full headline + blockers, run liveness panel (status, heartbeat age,
  orchestrator pass stats from `run.json`)
- PR table: per-check breakdown with links, resolvable/total thread counts,
  task counts, review decision, full stack context for the epic's PRs
- **history timeline**: every `history[]` entry (time, phase chip, note;
  `runId` linked when present); ack consumptions/supersessions appear here
- ack log (pending + recent), raw-JSON peek, "open state file" / "open epic
  file" links

## Click targets

| Target | Action |
|---|---|
| Waiting banner / "Open question" | `waitingOn.ref` in browser — deep link to the exact comment; the highest-value click in the product |
| Issue number / title | tracking issue in browser |
| PR row | PR in browser (setting to prefer the GitHub PRs extension) |
| CI badge | failing check's URL, else PR checks tab |
| Card body | expand (tray: in-place accordion; editor: right pane) |
| Repo badge | filter queue to repo |
| Restart run | configured per-repo command (`statusPipe.resumeCommand`, e.g. `claude -p '/epic-iteration-fanout'`) in the integrated terminal |

## Notifications

- **Activity-bar badge = NEEDS YOU count only.** Waiting/quiet churn never
  badges.
- Toasts, individually toggleable, all default-on: new blocker or owner
  question · agent crash / stale heartbeat / stale ack · item completed (quiet
  style) · CI flipped to failing with idle agent. Never for WAITING-lane churn.
  Global do-not-disturb setting.

## Deliberate omissions

No merge/approve buttons in the tray — those decisions belong on the forge page
where the full context lives (the alternative saves one click and invites
approving a stack you haven't looked at). No agent process management beyond
the restart-command passthrough. No stack graph beyond the up/down indicators —
the git-spice extension owns stack visualization.
